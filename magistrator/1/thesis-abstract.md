# Тезисы научно-исследовательской работы

## Интеллектуальная система агрегации, обработки и интерпретации потоковых данных для автоматизации принятия решений в киберфизических системах

---

## 1. К ПОСТАНОВКЕ ПРОБЛЕМЫ

### 1.1 Краткое вступление. Актуальность темы

Киберфизические системы (КФС) представляют собой интеграцию физических процессов и информационных систем для автоматизации промышленности, управления городской инфраструктурой, здравоохранения и других критических областей. Ключевой вызов современных КФС состоит в том, что они генерируют огромные объемы гетерогенных потоковых данных в реальном времени (скорость, объем, разнообразие). Традиционные системы обработки данных оказываются неэффективны для такого объема информации.

Актуальность работы определяется тремя факторами: **(1)** растущей сложностью КФС и требованиями к оперативности принятия решений, **(2)** несовершенством существующих подходов к агрегации гетерогенных потоков данных, **(3)** ограниченностью ресурсов на граничных вычислениях (edge computing), где должны приниматься локальные решения без задержки от облака.

### 1.2 Цель и задачи работы

**Цель работы**: разработать архитектуру интеллектуальной системы для агрегации, обработки и интерпретации потоковых данных КФС, позволяющей автоматизировать принятие решений в режиме реального времени с минимальной задержкой.

**Задачи 1-го года (текущий семестр)**:
- Провести анализ существующих методик и технологий обработки потоков данных в КФС
- Выявить главные недостатки текущих решений в контексте гибридной архитектуры (edge-cloud)
- Разработать концептуальные основы системы агрегации и интерпретации данных
- Определить приоритетные кейсы и сценарии использования

### 1.3 Обзор текущих методик решения задачи обработки потоковых данных

#### 1.3.1 Архитектура на основе облачных вычислений

Классический подход предполагает централизованную обработку всех данных в облаке (AWS IoT Core, Azure IoT Hub). 

**Достоинства**: масштабируемость, дешевизна хранилища, возможность использования мощных ML-алгоритмов.

**Недостатки**: высокая задержка сетевой передачи (неприемлемо для критичных по времени систем), высокая стоимость трафика, уязвимость к сбоям сети, проблемы с конфиденциальностью данных.

#### 1.3.2 Edge Computing и локальная обработка

Обработка данных на устройствах или граничных шлюзах (Kubernetes@Edge, Apache Kafka Edge, IoT gateways).

**Достоинства**: низкая задержка, снижение трафика, локальная автономность, конфиденциальность.

**Недостатки**: ограниченные вычислительные ресурсы, сложность синхронизации состояния между узлами, необходимость кастомизации для каждого устройства.

#### 1.3.3 Гибридные архитектуры (Edge-Cloud)

Комбинированный подход: реал-тайм аналитика на edge, обучение моделей и долгосрочный анализ в облаке (MQTT + Kafka + InfluxDB + ML).

**Достоинства**: баланс между низкой задержкой и масштабируемостью, интеллектуальная фильтрация данных.

**Недостатки**: сложность синхронизации, необходимость управления состоянием и версионированием, требует опыта в системной архитектуре.

#### 1.3.4 Технологии агрегации потоков

- **MQTT** (Message Queuing Telemetry Transport): легкий протокол для IoT, но не масштабируемый для больших потоков
- **Apache Kafka**: высокомасштабируемая платформа потоковой обработки, но требует больше ресурсов
- **Time-Series Databases** (InfluxDB, TimescaleDB): оптимизированы для хранения и запросов временных рядов
- **Stream Processing** (Apache Flink, Spark Streaming, ClickHouse): обработка в реальном времени с состоянием

#### 1.3.5 Методики интерпретации и принятия решений

- **Правила (Rule-based systems)**: быстро, но требуют ручной настройки
- **Machine Learning** (LSTM для временных рядов, изоляционные леса для аномалий): адаптивны, но требуют обучения
- **Комбинированные подходы**: гибридные системы с правилами + ML

### 1.4 Анализ достоинств и недостатков существующих методик

Основные выводы:

1. **Отсутствие единого стандарта**: каждый вендор предлагает собственные решения, интеграция сложна
2. **Проблема latency vs. scalability**: облако масштабируемо, но медленно; edge быстро, но ограничен по ресурсам
3. **Сложность управления гетерогенными данными**: разные типы датчиков, протоколы, форматы требуют унифицированного представления
4. **Недостаточная адаптивность**: большинство систем требуют ручной переконфигурации при изменении окружения
5. **Отсутствие тесной интеграции обработки и интерпретации**: данные обрабатываются отдельно, затем анализируются — теряется временной контекст

### 1.5 Вывод и направление последующего исследования

Существующие решения решают либо задачу масштабируемости (облако), либо задачу низкой задержки (edge), но редко обеспечивают оптимальный баланс и адаптивность к изменяющимся условиям. 

На 2-м году исследования планируется:
1. **Разработать архитектуру гибридной системы** с динамической балансировкой нагрузки между edge и cloud
2. **Предложить методику интеллектуальной фильтрации и агрегации** потоков на основе значимости данных
3. **Разработать алгоритм адаптивной интерпретации** потоков с использованием комбинированного подхода (правила + ML)
4. **Реализовать MVP** на практическом кейсе мониторинга окружающей среды и предиктивного обслуживания оборудования

Ожидаемые результаты: система, которая эффективнее на 30-40% справляется с задачей обработки потоков в условиях ограниченных сетевых ресурсов и обеспечивает принятие решений с задержкой < 500ms.
